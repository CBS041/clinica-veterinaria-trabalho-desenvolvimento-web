# AGENTS - Local developer agent guide

Purpose: Quickly orient an AI coding agent to this small Java CLI project so it can be productive immediately.

1) Big picture
- Minimal MVC-ish CLI app: `view` contains console UI (`UsuarioView`, `TutorView`, `AnimalView`), `controller` contains business logic (`UsuarioController`, `TutorController`, `AnimalController`), and `model` holds domain types (`Usuario`, `Cliente`, `Administrador`, `Pessoa`, `Tutor`, `Animal`, `Cachorro`, `Gato`).
- Main entry: `src/Main.java` instantiates `TutorController` and `AnimalController`, wires `TutorView` and `AnimalView`, and shows a top-level module menu. `UsuarioView`/`UsuarioController` still exist as a separate user-management flow but are not currently invoked from `Main`.

2) Key architectural facts an agent must know
- Data is in-memory only: `UsuarioController`, `TutorController`, and `AnimalController` each use a private `List<...>` plus an `int contadorId` as an auto-increment id. No database or external services.
- Models: `Usuario` is abstract and defines common fields and `getTipoUsuario()`; `Cliente` and `Administrador` extend it. `Pessoa` is the shared base for `Tutor`, and `Animal` is abstract with concrete `Cachorro` and `Gato` subclasses. Display still relies on `toString()` in the concrete classes.
- View owns input parsing and simple validation; `AnimalView` also resolves the tutor by ID through `TutorController.buscarPorId(...)` before creating `Cachorro`/`Gato` and tolerates a missing tutor by storing `null`.
